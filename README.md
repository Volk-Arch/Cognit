# Cognit — Persistent Neural Context

> **TL;DR для тех кто в теме:**
> Сохраняем KV-cache Transformer и рекуррентное состояние RWKV на диск — это и есть "паттерны".
> Агенты реализованы через eval дополнительного контента поверх базового KV-cache без `save_state` — ephemeral merge.
> RWKV-состояние фиксированного размера (~98 KB для 7B) вне зависимости от объёма входного текста — отсюда безлимитный индекс всей кодовой базы.
> Переключение моделей in-process через возврат handoff-dict из `cli_loop()`, без subprocess.

## Как это работает

Нейросеть один раз читает файлы и **сохраняет своё внутреннее состояние** на диск. Все последующие вопросы идут к уже «знающей» модели — файлы не передаются повторно.

Это не RAG и не векторный поиск. Модель держит код в памяти так, как держит его человек — целиком, с пониманием связей между модулями.

В системе **две модели** с разными ролями:

| | RWKV | Transformer |
|---|---|---|
| Роль | Глобальный индекс | Локальный анализ |
| Контекст | **Без ограничений** | 8192 токенов |
| Паттерн | ~98 KB (фиксированный) | Растёт с диалогом |
| Сильная сторона | Весь проект целиком | Точная работа с файлом |

Модели **передают задачи друг другу** — переключение происходит в одном процессе без перезапуска: RWKV находит нужные файлы → Transformer подхватывает их. Transformer упирается в контекст → RWKV берёт задачу шире. Единая точка входа — `cognit.py`.

**Паттерны** (сохранённые состояния) привязаны к git-ветке — переключился на другую ветку, Cognit автоматически использует её паттерны. При `git commit` паттерны изменённых файлов обновляются через post-commit хук.

## Сценарии использования

**Разобраться в чужом проекте** — RWKV читает всю кодовую базу за один проход, отвечает на вопросы об архитектуре без повторной передачи файлов.

**Детальный анализ и правки** — Transformer работает с конкретным файлом, накапливает диалог. `/patch` применяет предложенные изменения прямо из ответа модели.

**Писать код с учётом стиля** — `/agent style` включает ambient-режим: каждый ответ модели автоматически проходит через знания агента. Стиль учитывается при написании, а не при ревью.

**Ревью по соглашениям команды** — `/review @file` стекает знание агента + файл в один вопрос. Паттерн не меняется — эфемерная сессия.

**Найти нужные файлы для задачи** — `route` в RWKV определяет, какие файлы затронет задача. Transformer получает только их, без лишнего контекста.

**База знаний команды** — `agents/` с описанием архитектуры, стиля, контекста хранится в git и загружается при каждой сессии.

## Установка и настройка

```bash
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu121
```

Скачать модели (GGUF) и положить в `models/`:
- **Transformer**: `Qwen3-8B-Q4_K_M.gguf` (~4.7 GB VRAM)
- **RWKV**: `models/rwkv/RWKV-6-World-7B-Q4_K_M.gguf` (~4.1 GB VRAM)

> Обе модели не нужны одновременно — RWKV выгружается из VRAM перед стартом Transformer.

Одноразовая настройка — post-commit хук + `agents/` + `.echo.json`:

```bash
python cognit_setup.py
```

Спросит путь к клиентскому проекту, установит хук туда и создаст шаблоны `agents/`.

## Быстрый старт

### Что загружать в какую модель

| | RWKV (`--rwkv`) | Transformer (по умолчанию) |
|---|---|---|
| **Загружай** | Вся кодовая база, целый модуль, много файлов | Конкретный файл, который будешь менять |
| **Используй для** | «В каком файле X?», «Какие файлы затронет задача?» | «Есть ли баги?», «Добавь валидацию», `/patch` |
| **Запуск** | `python cognit.py --rwkv` | `python cognit.py` |

Паттерн создаётся в той модели, которая загружена при `/load`. Выбор — при **запуске**, не при загрузке.

---

### Сценарий 1: знаешь файл → сразу Transformer

```
python cognit.py

🧠> /load auth @src/auth/middleware.py
🧠 [auth]> есть ли уязвимости в JWT-проверке?
🧠 [auth]> исправь алгоритм
🧠 [auth]> /patch
```

`/patch` — если в ответе нет diff-блока, автоматически запрашивает его у модели, показывает изменения и применяет с созданием `.cognit.bak` бэкапа.

### Сценарий 2: не знаешь файл → route прямо из Transformer

Если RWKV-индекс уже создан (`/load repo @src/` в RWKV-режиме):

```
python cognit.py

🧠> route добавить rate limiting для POST /login

   Найден RWKV-индекс: repo
⚙️  Переключение: Transformer → RWKV  (авто-маршрут: добавить rate limiting...)
⚡ Авто-маршрут: «добавить rate limiting для POST /login»
   Индекс: repo

→ Файлы: middleware.py, routes.py, db/connection.py

⚙️  Переключение: RWKV → Transformer  (файлов: 3)

🧠 [middleware]> как сейчас реализована ограничение запросов?
```

Если RWKV-индекса нет — понятное сообщение об ошибке с инструкцией.
Создать индекс (один раз): `python cognit.py --rwkv` → `/load repo @src/`

Переключение происходит в одном процессе — без перезапуска, без потери сессии.

### Сценарий 3: начал в Transformer, понял что мало контекста → expand

```
python cognit.py

🧠> /load auth @src/auth/middleware.py
🧠 [auth]> как это соотносится с остальной авторизацией?
🧠 [auth]> expand нужен контекст по всей авторизации

⚙️  Переключение: Transformer → RWKV
🧠> use repo
🧠 [repo]> расскажи об архитектуре авторизации в проекте
```

### Персистентность

```bash
# Завершили сессию, перезапустили
python cognit_transformer.py
📋 Паттернов: 3
🧠> use auth
🧠 [auth]> продолжим — что ещё проверить?
```

Файл не перечитывается — только вопрос.

## Техническая информация

### Оркестратор

`cognit.py` — единая точка входа. Запускает нужный бэкенд по `.echo.json` или флагу, управляет переключением в одном процессе:

| Команда | Где вводить | Эффект |
|---|---|---|
| `expand <задача>` | Transformer | Transformer выгружается, RWKV загружается с задачей |
| `route <задача>` | Transformer | RWKV авто-запускается, находит файлы, Transformer загружает их |
| `route <задача>` | RWKV | RWKV находит файлы → Transformer загружается с ними |

Каждый бэкенд можно запускать и напрямую (`cognit_transformer.py`, `cognit_rwkv.py`) — оркестратор не обязателен.

### Правка кода (/patch)

Transformer предложил изменения → `/patch` извлекает unified diff из ответа, показывает его и применяет к файлу. Файл определяется автоматически из заголовка `+++ b/path` diff-блока. Перед записью создаётся `.cognit.bak`.

Если в ответе нет diff-блока — `/patch` автоматически переспрашивает модель.

### Хранение паттернов

```
echo_patterns/
  <repo>/
    <branch>/
      <name>.pkl   ← состояние нейросети (KV-cache или RWKV state)
      <name>.json  ← метаданные: файлы-источники, хэши, дата, диалоги
```

### Политика обновления (grow_policy)

| Политика | Условие | Поведение |
|---|---|---|
| `retrain` | `main`/`master`, `agents/` | Пересоздаётся при коммите |
| `grow ~` | Feature-ветки | Пропускается хуком, накапливает диалог |

```
🧠 [auth]>   ← retrain  🧠 [auth~]>  ← grow
```

- `/load ?name @path` — форсировать `retrain` принудительно
- `? вопрос` — временно сменить политику на один вопрос (grow↔retrain)

### agents/ — база знаний проекта

Markdown-документы со спецификой проекта, хранятся в git клиентского проекта:

```
agents/
  style/global.md      ← именование, форматирование, запреты
  arch/overview.md     ← модули, зависимости, поток запроса
  context/project.md   ← цель проекта, бизнес-правила
```

```bash
python cognit_setup.py agents  # создать шаблоны
```

Два режима работы с агентами:

| Режим | Команда | Когда |
|---|---|---|
| **Ephemeral** | `/review [агент] @file` | Разовая проверка, паттерн не меняется |
| **Ambient** | `/agent <имя> [имя2 ...]` | Каждый вопрос учитывает агентов, тоже эфемерно |

```
🧠 [auth]> /agent style arch        ← включить ambient: стиль + архитектура
🧠 [auth + style + arch]> добавь rate limiting   ← ответ учитывает оба агента
🧠 [auth + style + arch]> /agent off  ← выключить
🧠 [auth]>
```

### Git-интеграция

Post-commit хук обновляет паттерны при коммите. Паттерны с `grow` пропускаются (накапливают диалог). Ручное управление:

```bash
python cognit_transformer.py --refresh-file src/auth.py  # пересоздать паттерн
python cognit_transformer.py --status                    # показать устаревшие
```

### Конфигурация

| Параметр | Transformer | RWKV |
|---|---|---|
| `N_CTX` | 8192 токенов | 1024 (буфер генерации) |
| `N_GPU_LAYERS` | -1 (все на GPU) | -1 |
| `MAX_TOKENS` | 512 | 512 |
| `CHUNK_SIZE` | — | 512 |

### Структура проектов

```
my-project/              ← клиентский git (твой код)
  agents/                ← база знаний (в git!)

Cognit/                  ← этот репо (рядом)
  cognit.py              ← оркестратор (единая точка входа)
  cognit_transformer.py  ← Transformer-бэкенд (KV-cache, Qwen3)
  cognit_rwkv.py         ← RWKV-бэкенд (рекуррентное состояние)
  cognit_core.py         ← общие утилиты (git, паттерны, хэши)
  cognit_patch.py        ← применение unified diff
  cognit_agents.py       ← работа с agents/ (чтение, список)
  cognit_setup.py        ← одноразовая настройка
  echo_patterns/         ← паттерны (в .gitignore)
  models/                ← GGUF-файлы (в .gitignore)
  .echo.json             ← путь к клиентскому проекту (в .gitignore)
```

⚠️ При переносе Cognit в другую папку — перезапустить `python cognit_setup.py`.

## Ограничения

### Контекст Transformer ограничен 8192 токенами

Это ~300–400 строк кода. Большой файл придётся обрезать или загружать по частям. Длинный диалог в grow-паттерне постепенно вытесняет исходный контекст — модель начинает «забывать» детали файла.

### RWKV теряет детали при сжатии

Рекуррентное состояние — фиксированного размера (~98 KB). При обработке большой кодовой базы ранние файлы постепенно вытесняются поздними. Модель «видела» весь код, но не удерживает все детали с одинаковой точностью. Для маршрутизации (`route`) это обычно достаточно; для детального анализа — уже нет.

### Качество зависит от локальной модели

Qwen3-8B и RWKV-7B — модели среднего класса. Ответы будут хуже, чем у GPT-4 или Claude при прямой передаче файла в контекст. Cognit не улучшает модель — он убирает необходимость каждый раз передавать контекст заново.

### `route` может возвращать неточные пути

RWKV определяет нужные файлы по запросу на естественном языке. Модель может назвать несуществующие пути или пропустить нужный файл — особенно если кодовая база большая и имена файлов неочевидны. Список файлов всегда показывается пользователю перед загрузкой.

### Паттерны не переносимы между машинами (пока)

`echo_patterns/` содержит бинарные состояния модели — они привязаны к конкретной версии модели и llama-cpp-python. Перенос на другую машину или обновление библиотеки потребует пересоздания паттернов. S3-синхронизация запланирована, но не реализована.

### Proof of concept, не production-инструмент

Нет тестов, нет обработки всех краевых случаев, CLI-интерфейс требует привычки. Проект показывает идею — персистентный нейронный контекст как альтернатива RAG — и служит основой для дальнейшего развития.

## Шпаргалка

```bash
# Запуск
python cognit.py               # единая точка входа (из .echo.json)
python cognit.py --rwkv        # принудительно RWKV
python cognit.py --transformer # принудительно Transformer
python cognit_transformer.py   # Transformer напрямую
python cognit_rwkv.py          # RWKV напрямую

# Паттерны  [обе модели]
/load name @path               # загрузить файл или папку
/load ?name @path              # загрузить с принудительным retrain
/load name текст               # загрузить текст напрямую
use name                       # переключиться на паттерн
/list                          # список паттернов (~ = grow)

# Вопросы  [обе модели]
вопрос                         # ask (следует политике паттерна)
? вопрос                       # временно сменить политику (grow↔retrain)

# Правка кода  [Transformer]
/patch                         # diff из последнего ответа → применить к файлу
/patch @file                   # применить к конкретному файлу

# Агенты  [Transformer]
/agents                        # список доступных агентов из agents/
/agent name [name2 ...]        # ambient: все вопросы через [паттерн + агенты]
/agent off                     # выключить ambient
/review @file                  # разовое ревью файла (агент style, эфемерно)
/review agent @file            # ревью через конкретный агент
/review                        # ревью последнего ответа модели
/style @file                   # проверка стиля файла

# Передача между моделями
expand задача                  # [Transformer] передать задачу в RWKV (обзор архитектуры)
route задача                   # [Transformer] найти файлы через RWKV-индекс → загрузить их
route задача                   # [RWKV] найти файлы для задачи → handoff на Transformer

# Headless
python cognit_transformer.py --status               # проверить устаревшие паттерны
python cognit_transformer.py --refresh-file path    # пересоздать паттерн вручную
python cognit_rwkv.py --auto-expand                 # принять pending задачу от Transformer
python cognit_setup.py                              # перенастроить после смены папки
python cognit_setup.py agents                       # создать agents/ в клиентском проекте
```
