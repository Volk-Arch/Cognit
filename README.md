# Cognit — Persistent Neural Context

> **TL;DR для тех кто в теме:**
> Сохраняем KV-cache Transformer и рекуррентное состояние RWKV на диск — это и есть "паттерны".
> Агенты реализованы через eval дополнительного контента поверх базового KV-cache без `save_state` — ephemeral merge.
> RWKV-состояние фиксированного размера (~98 KB для 7B) вне зависимости от объёма входного текста — отсюда безлимитный индекс всей кодовой базы.
> Переключение моделей in-process через возврат handoff-dict из `cli_loop()`, без subprocess.

## Как это работает

Нейросеть один раз читает файлы и **сохраняет своё внутреннее состояние** на диск. Все последующие вопросы идут к уже «знающей» модели — файлы не передаются повторно.

Это не RAG и не векторный поиск. Модель держит код в памяти так, как держит его человек — целиком, с пониманием связей между модулями.

Система работает **последовательно на одной видеокарте** — модели не запускаются параллельно, при переключении одна выгружается из VRAM, другая загружается. Это позволяет работать даже на 6–8 GB VRAM и не требует дорогого оборудования.

В системе **две модели** с разными ролями:

| | RWKV | Transformer |
|---|---|---|
| Роль | Глобальный индекс | Локальный анализ |
| Контекст | **Без ограничений** | 8192 токенов |
| Паттерн | ~98 KB (фиксированный) | Растёт с диалогом |
| Сильная сторона | Весь проект целиком | Точная работа с файлом |

**Пользователь всегда общается с Transformer.** RWKV — внутренний инструмент: запускается автоматически для индексирования и маршрутизации, CLI не показывает. Единая точка входа — `cognit.py`.

Когда Transformer встречает `route` или `expand` в запросе — RWKV запускается headless в том же процессе: строит индекс (если нет), находит нужные файлы, возвращает их Transformer'у. Пользователь видит только сообщение «⚙️ RWKV: маршрутизирую...» и итоговый ответ.

**Паттерны** (сохранённые состояния) привязаны к git-ветке — переключился на другую ветку, Cognit автоматически использует её паттерны. При `git commit` паттерны изменённых файлов обновляются через post-commit хук.

## Сценарии использования

**Разобраться в чужом проекте** — RWKV читает всю кодовую базу за один проход, отвечает на вопросы об архитектуре без повторной передачи файлов.

**Детальный анализ и правки** — Transformer работает с конкретным файлом, накапливает диалог. `/patch` применяет предложенные изменения прямо из ответа модели.

**Писать код с учётом стиля** — `/agent style` включает ambient-режим: каждый ответ модели автоматически проходит через знания агента. Стиль учитывается при написании, а не при ревью.

**Ревью по соглашениям команды** — `/review @file` стекает знание агента + файл в один вопрос. Паттерн не меняется — эфемерная сессия.

**Найти нужные файлы для задачи** — `route <задача>` в Transformer запускает RWKV headless: он определяет нужные файлы, Transformer подхватывает их автоматически и сразу отвечает. Пользователю не нужно переключаться вручную.

**База знаний команды** — `agents/` с описанием архитектуры, стиля, контекста хранится в git и загружается при каждой сессии.

**Агенты без настройки** — при старте Cognit проверяет `agents/` и автоматически создаёт паттерны для агентов, у которых ещё нет `.pkl`. При каждом вопросе активирует нужных агентов по ключевым словам — спросил про стиль, включился `style`; про архитектуру — `arch`. Ничего настраивать не нужно.

## Установка и настройка

```bash
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu121
```

Скачать модели (GGUF) и положить в `models/`:
- **Transformer**: `Qwen3-8B-Q4_K_M.gguf` (~4.7 GB VRAM) — или любой Qwen/Mistral/Llama GGUF
- **RWKV**: `RWKV-6-World-7B-Q4_K_M.gguf` (~4.1 GB VRAM) — или 3B (~2 GB) / 1B6 (~1 GB)

> Обе модели не нужны одновременно — RWKV выгружается из VRAM перед стартом Transformer. Весь процесс работает последовательно, на одной видеокарте.

Одноразовая настройка:

```bash
python cognit_setup.py
```

Спросит путь к клиентскому проекту и пути к моделям, установит git-хуки и создаст шаблоны `agents/`.

Шаблоны агентов уже заполнены реальным содержимым — `style/global.md` содержит Python-конвенции, `arch/overview.md` — трёхслойную структуру, `context/project.md` — шаблон с примерами. Работает сразу без правок.

### Конфигурация (`echo.json`)

Все настройки хранятся в `.echo.json`. Генерируется `cognit_setup.py`, редактируется вручную:

```json
{
  "backend": "transformer",
  "transformer": {
    "model_path":   "models/Qwen3-8B-GGUF/Qwen3-8B-Q4_K_M.gguf",
    "n_gpu_layers": -1,
    "n_ctx":        8192,
    "max_tokens":   512
  },
  "rwkv": {
    "model_path":   "models/rwkv/RWKV-6-World-7B-Q4_K_M.gguf",
    "n_gpu_layers": -1,
    "n_ctx":        1024,
    "max_tokens":   512,
    "chunk_size":   512
  },
  "client_project": "C:/path/to/your/project"
}
```

`n_gpu_layers: -1` — все слои на GPU. `0` — CPU-only. Оба бэкенда читают свою секцию при старте.

## Быстрый старт

### Типичная сессия

Запуск всегда один — `python cognit.py`. RWKV включается сам, когда нужен:

| Что вводишь | Что происходит |
|---|---|
| `/load auth @src/auth.py` + вопрос | Transformer читает файл, отвечает |
| `route добавить rate limiting` | ⚙️ RWKV headless: находит файлы → Transformer загружает и отвечает |
| `expand покажи всю архитектуру` | ⚙️ RWKV headless: индексирует проект → маршрутизирует → Transformer отвечает |
| `/edit @src/auth.py убрать дублирование` | Transformer читает файл свежим, генерирует unified diff |
| `/patch` | Применяет diff из последнего ответа к файлу |
| `python cognit.py --rwkv` | Интерактивный RWKV (только для ручного построения индекса) |

Пошаговый сценарий установки и примеры работы: [TESTING.md](TESTING.md)

## Техническая информация

### Оркестратор

`cognit.py` — единая точка входа. Пользователь **всегда** в Transformer. RWKV запускается headless (без CLI) автоматически — пользователь его не видит.

| Команда в Transformer | Эффект |
|---|---|
| `route <задача>` | ⚙️ RWKV headless: маршрутизирует через готовый индекс → Transformer загружает файлы и сразу отвечает |
| `expand <задача>` | ⚙️ RWKV headless: строит индекс проекта (если нет) → маршрутизирует → Transformer берёт задачу |
| `python cognit.py --rwkv` | Интерактивный RWKV CLI (только для ручного `/load repo @src/`) |

При `route`/`expand` переключение прозрачно: Transformer выгружается → RWKV запускается headless → выгружается → Transformer загружается снова. Пользователь видит только `⚙️ RWKV: маршрутизирую...` и итоговый ответ.

Каждый бэкенд запускается и напрямую (`cognit_transformer.py`, `cognit_rwkv.py`) — оркестратор не обязателен.

### Правка кода (/patch и /edit)

**`/patch`** — Transformer предложил изменения → извлекает unified diff из последнего ответа, показывает и применяет к файлу. Файл определяется из заголовка `+++ b/path`. Перед записью создаётся `.cognit.bak`. Если diff нет — модель переспрашивается автоматически.

**`/edit @file задача`** — для точных правок: читает файл заново (не из KV-cache), передаёт содержимое в модель, просит unified diff. Используй когда нужна точная правка конкретного места, а не ответ по памяти из паттерна.

### Хранение паттернов

```
echo_patterns/
  <repo>/
    <branch>/
      <name>.pkl   ← состояние нейросети (KV-cache или RWKV state)
      <name>.json  ← метаданные: файлы-источники, хэши, дата, диалоги
```

### Политика обновления (grow_policy)

| Политика | Условие | Поведение |
|---|---|---|
| `retrain` | `main`/`master`, `agents/` | Пересоздаётся при коммите |
| `grow ~` | Feature-ветки | Пропускается хуком, накапливает диалог |

```
🧠 [auth]>   ← retrain  🧠 [auth~]>  ← grow
```

- `/load ?name @path` — форсировать `retrain` принудительно
- `? вопрос` — временно сменить политику на один вопрос (grow↔retrain)

### agents/ — база знаний проекта

Markdown-документы со спецификой проекта, хранятся в git клиентского проекта:

```
agents/
  style/global.md      ← именование, форматирование, запреты
  arch/overview.md     ← модули, зависимости, поток запроса
  context/project.md   ← цель проекта, бизнес-правила
```

```bash
python cognit_setup.py agents  # создать шаблоны
```

**Авто-инициализация** — при запуске Cognit сканирует `agents/` клиентского проекта. Если паттерн агента ещё не создан (нет `.pkl`) — создаётся автоматически. Пользователь ничего не настраивает вручную.

**Авто-выбор агентов** — если ambient не задан вручную через `/agent`, Cognit при каждом вопросе автоматически активирует нужных агентов. Два механизма:

1. **Оркестратор (основной)** — нейронный роутинг через `_peek_pattern`: тихий инференс «какие агенты нужны для этого вопроса?» → список → активация. Требует хотя бы одного агента с `card.md`.

2. **Ключевые слова (fallback)** — если оркестратора нет, Cognit использует таблицу ключевых слов:

| Агент | Активируется если вопрос содержит |
|---|---|
| `style` | стиль, форматирование, naming, lint, pep8, отступ... |
| `arch` | архитектур, структур, design, паттерн, зависимост, слой... |
| `context` | контекст, проект, цель, требовани, бизнес, зачем, почему... |

Напечатает `🤖 Авто-агенты: style` — пользователь видит что происходит. Агентам можно добавить `keywords.txt` в папку агента для кастомных ключевых слов.

Три режима работы с агентами:

| Режим | Команда | Когда |
|---|---|---|
| **Авто** | — | Ключевые слова → Cognit сам активирует нужных |
| **Ephemeral** | `/review [агент] @file` | Разовая проверка, паттерн не меняется |
| **Ambient** | `/agent <имя> [имя2 ...]` | Каждый вопрос учитывает агентов, тоже эфемерно |

```
🧠 [auth]> добавь rate limiting        ← Cognit замечает «добавь» + auth тематика
🤖 Авто-агенты: style arch             ← сам включил два агента
🧠 [auth]> /agent style arch           ← или вручную: всегда эти агенты
🧠 [auth + style + arch]> добавь rate limiting
🧠 [auth + style + arch]> /agent off   ← выключить
🧠 [auth]>
```

**Цепочки по ответу агента** — если модель в ответе предложила `expand` или `route`, Cognit замечает это и автоматически запускает нужный headless-бэкенд без вопросов к пользователю.

**Оркестратор-паттерн** — нейронный роутер вместо keyword-matching. При старте Cognit генерирует `card.md` для каждого агента через его собственный KV-cache — модель описывает себя: «что я знаю, когда меня надо». Все `card.md` объединяются в паттерн `orchestrator`. Перед каждым вопросом тихий инференс: «какие агенты нужны?» → список → активация.

```
agents/
  style/
    global.md        ← знания агента (заполняешь ты)
    card.md          ← авто-генерируется из KV-cache: "Область: стиль кода..."
  arch/
    overview.md
    card.md          ← авто-генерируется
```

Полностью автоматический цикл: заполнил `global.md` → при следующем запуске модель читает его, создаёт паттерн, генерирует своё описание, строит оркестратор. Fallback на keyword-matching если оркестратора нет.

### Git-интеграция

`cognit_setup.py` устанавливает два хука в клиентский проект:

**post-commit** — при каждом коммите обновляет паттерны изменённых файлов. Паттерны с `grow` пропускаются (накапливают диалог).

**post-checkout** — при смене ветки проверяет наличие RWKV-индекса для новой ветки. Если индекса нет — предупреждает и подсказывает как создать. Хук не загружает модель — работает мгновенно.

```bash
python cognit_transformer.py --refresh-file src/auth.py  # пересоздать паттерн вручную
python cognit_transformer.py --status                    # показать устаревшие паттерны
python cognit_rwkv.py --check-index                      # есть ли RWKV-индекс (без загрузки)
python cognit_rwkv.py --build-index src/                 # создать RWKV-индекс headless
```

### Параметры по умолчанию

| Параметр | Transformer | RWKV |
|---|---|---|
| `n_ctx` | 8192 токенов | 1024 (буфер генерации) |
| `n_gpu_layers` | -1 (все на GPU) | -1 |
| `max_tokens` | 512 | 512 |
| `chunk_size` | — | 512 |

Все параметры переопределяются в `.echo.json` — редактировать Python-файлы не нужно.

### Структура проектов

```
my-project/              ← клиентский git (твой код)
  agents/                ← база знаний (в git!)

Cognit/                  ← этот репо (рядом)
  cognit.py              ← оркестратор (единая точка входа)
  cognit_transformer.py  ← Transformer-бэкенд (KV-cache, Qwen3)
  cognit_rwkv.py         ← RWKV-бэкенд (рекуррентное состояние)
  cognit_core.py         ← общие утилиты (git, паттерны, хэши)
  cognit_patch.py        ← применение unified diff
  cognit_agents.py       ← работа с agents/ (чтение, список)
  cognit_setup.py        ← одноразовая настройка
  echo_patterns/         ← паттерны (в .gitignore)
  models/                ← GGUF-файлы (в .gitignore)
  .echo.json             ← путь к клиентскому проекту (в .gitignore)
```

⚠️ При переносе Cognit в другую папку — перезапустить `python cognit_setup.py`.

## Ограничения

### Контекст Transformer ограничен 8192 токенами

Это ~300–400 строк кода. Большой файл придётся обрезать или загружать по частям. Длинный диалог в grow-паттерне постепенно вытесняет исходный контекст — модель начинает «забывать» детали файла.

### RWKV теряет детали при сжатии

Рекуррентное состояние — фиксированного размера (~98 KB). При обработке большой кодовой базы ранние файлы постепенно вытесняются поздними. Модель «видела» весь код, но не удерживает все детали с одинаковой точностью. Для маршрутизации (`route`) это обычно достаточно; для детального анализа — уже нет.

### Качество зависит от локальной модели

Qwen3-8B и RWKV-7B — модели среднего класса. Ответы будут хуже, чем у GPT-4 или Claude при прямой передаче файла в контекст. Cognit не улучшает модель — он убирает необходимость каждый раз передавать контекст заново.

### `route` может возвращать неточные пути

RWKV определяет нужные файлы по запросу на естественном языке. Модель может назвать несуществующие пути или пропустить нужный файл — особенно если кодовая база большая и имена файлов неочевидны. Список файлов всегда показывается пользователю перед загрузкой.

### Паттерны не переносимы между машинами (пока)

`echo_patterns/` содержит бинарные состояния модели — они привязаны к конкретной версии модели и llama-cpp-python. Перенос на другую машину или обновление библиотеки потребует пересоздания паттернов. S3-синхронизация запланирована, но не реализована.

### CLI-интерфейс требует привычки

Нет графического интерфейса — только терминал. Web UI запланирован. Не все краевые случаи обработаны. Проект активно развивается.

## Шпаргалка

**Типичный сеанс — файл известен:**
```
python cognit.py
🧠> /load auth @src/auth.py   ← читаем файл
🧠 [auth]> есть ли баги?      ← вопрос (паттерн растёт)
🧠 [auth]> исправь            ← модель предлагает diff
🧠 [auth]> /patch             ← применяем к файлу
🧠 [auth]> /exit
```

**Файл неизвестен — zero-touch:**
```
python cognit.py
🧠> route добавить логирование в auth-модуль
   ⚙️  RWKV: маршрутизирую...           ← RWKV запустился headless
→ Файлы: src/auth.py, src/middleware.py
🧠 [auth]> где именно добавить?          ← Transformer уже загрузил файлы и ответил
```

```bash
# Запуск
python cognit.py               # единая точка входа  →  всегда Transformer
python cognit.py --rwkv        # интерактивный RWKV (ручное построение индекса)
python cognit.py --transformer # явно Transformer
python cognit_transformer.py   # Transformer напрямую (без оркестратора)
python cognit_rwkv.py          # RWKV напрямую

# Паттерны  [Transformer]
/load auth @src/auth.py        # загрузить файл → паттерн 'auth'
/load repo @src/               # загрузить всю папку
/load mix @src/ @docs/         # composite: два источника, один eval-проход
/load ?auth @src/auth.py       # загрузить с принудительным retrain (? = force)
/load note Не забыть: X        # загрузить произвольный текст
use auth                       # переключиться на паттерн 'auth'
/list                          # список паттернов  (auth~ = grow, auth = retrain)
/exit                          # выход

# Вопросы  [Transformer]
как работает функция login?    # ask — паттерн растёт (если grow) или не меняется (retrain)
? как работает функция login?  # peek — однократно инвертирует политику:
                               #   grow-паттерн → вопрос без сохранения
                               #   retrain-паттерн → вопрос с сохранением

# Правка кода  [Transformer]
/patch                         # diff из последнего ответа → показать → применить к файлу
/patch @src/other.py           # применить к конкретному файлу (переопределяет +++ заголовок)
                               # если diff нет — модель переспрашивается автоматически
/edit @src/auth.py убери дублирование  # читает файл свежим → просит unified diff напрямую

# Агенты  [Transformer]
/agents                        # список агентов из agents/ (и загружены ли их паттерны)
/agent style arch              # ambient: каждый вопрос через [auth + style + arch]
/agent off                     # выключить ambient → обычный режим
/review @src/auth.py           # разовое ревью файла через style-агент (паттерн не меняется)
/review arch @src/auth.py      # ревью через конкретный агент
/review                        # ревью последнего ответа модели
/style @src/auth.py            # проверка стиля файла
```
