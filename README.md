# Echo PoC — Persistent Neural Context

Локальная LLM с персистентной памятью. Модель один раз читает текст — состояние сохраняется в файл. Все последующие вопросы задаются без повторной передачи контекста.

## Два режима

| | `echo_poc.py` | `echo_rwkv.py` |
|---|---|---|
| Архитектура | Transformer (Qwen3, Phi, ...) | RWKV |
| Хранение памяти | KV-cache | Рекуррентное состояние |
| Лимит контекста | 8192 токенов | **Без ограничений** |
| Точность на деталях | Высокая | Чуть ниже (сжатие) |
| Команда `route` | — | ✅ (маршрутизация файлов) |
| Когда использовать | Отдельные файлы, точные вопросы | Большие кодовые базы, глобальный индекс |

**Гибридный рабочий процесс:**
RWKV индексирует всю кодовую базу → `route` определяет нужные файлы → Transformer работает с ними точно.

Паттерны обоих режимов хранятся в одной папке `echo_patterns/` и **несовместимы** между собой — паттерн привязан к модели, которой создан.

## Требования

```
pip install llama-cpp-python --extra-index-url https://abetlen.github.io/llama-cpp-python/whl/cu121
```

**Transformer** — модели в `models/`, указать путь в `echo_poc.py`:
- `Qwen3-8B-Q4_K_M.gguf` — ~4.7 GB VRAM (рекомендуется)
- `Phi-3.5-mini-instruct-Q4_K_M.gguf` — ~2.5 GB VRAM

**RWKV** — модели в `models/rwkv/`, указать путь в `echo_rwkv.py`:
- `RWKV-6-World-7B-Q4_K_M.gguf` — ~4.1 GB VRAM (рекомендуется)
- `RWKV-6-World-3B-Q4_K_M.gguf` — ~2.0 GB VRAM
- `RWKV-6-World-1B6-Q4_K_M.gguf` — ~1.0 GB VRAM

## Команды

Интерфейс одинаков для обоих режимов. Нужно запомнить 4 вещи:

```
use <имя>            — выбрать активный паттерн
<вопрос>             — спросить (паттерн растёт)
? <вопрос>           — спросить без изменения паттерна
/load <имя> @<путь>  — загрузить файл или папку
```

### Типичная сессия

```
# Запуск
python echo_poc.py

# Загрузка файла (@ = путь к файлу или папке)
🧠> /load auth @src/auth.py

# Переключение на паттерн (автоматически после /load)
🧠 [auth]> _

# Просто пишешь вопрос — это ask (паттерн растёт)
🧠 [auth]> что делает функция login?

# ? в начале — peek (паттерн не меняется)
🧠 [auth]> ? есть ли SQL-инъекции?

# Сменить паттерн
🧠 [auth]> use api
🧠 [api]> _
```

---

### `use <имя>`

Выбрать паттерн как активный. Все последующие вопросы идут к нему.

```
use auth
use style        ← переключиться на паттерн стиля кода
```

---

### `<вопрос>` — ask (grow)

Просто пишешь вопрос — модель отвечает из памяти паттерна.
После ответа паттерн **обновляется** — диалог накапливается.

---

### `? <вопрос>` — peek (no grow)

Ответ без изменения паттерна. Для разовых вопросов к фиксированному снимку.

```
🧠 [auth]> ? а если передать None вместо пароля?
```

---

### `/load <имя> @<путь>`

Загрузить файл или **целую папку** как паттерн. После загрузки паттерн активируется автоматически.

```
/load auth @src/auth.py          ← файл
/load style @agents/style/       ← вся папка (.md, .txt)
/load repo @src/                 ← кодовая база (RWKV, без лимита)
```

Директория: все `.md` и `.txt` файлы внутри конкатенируются в один паттерн. Пути к файлам сохраняются в метаданных — паттерн следит за изменениями.

Можно и текст напрямую:
```
/load note Важно: не использовать глобальные переменные
```

---

### `route <задача>`  *(только echo_rwkv.py)*

Спрашивает **активный** RWKV-паттерн, какие файлы нужны для задачи.
Выводит готовые `/load` команды для Transformer.

```
🧠 [repo]> route добавить rate limiting в API

   Команды для echo_poc.py:
   /load middleware @src/auth/middleware.py
   /load api_routes @src/api/routes.py
```

**Сценарий:**
1. `echo_rwkv.py` → `/load repo @src/` → `use repo`
2. `route добавить логин` → список файлов
3. Скопируй `/load` команды в `echo_poc.py` → работай точно

---

### `/list` / `/help` / `/exit`

Список паттернов, справка, выход.

## Структура проектов

### Два репозитория

Рекомендуемая организация: **один AI-репозиторий на основной проект**.

```
my-project/          ← основной гит (твой код)
  src/
  .echo.json         ← только ссылка на AI-проект
  .gitignore

my-project-ai/       ← AI-гит (рядом)
  agents/            ← знания о проекте (в git!)
    style/
      global.md
      commands.md
    arch/
      overview.md
    context/
      project.md
  echo_patterns/     ← нейросетевые состояния (в .gitignore)
  echo_poc.py
  echo_rwkv.py
  echo_setup.py
  models/            ← GGUF-файлы
```

Основной проект не знает о нейросетях — только `.echo.json` с путём к AI-проекту.
AI-проект — отдельный гит. Его можно шарить между членами команды через S3.

## agents/ — знания о проекте

`agents/` — папка с Markdown-документами, которые "обучают" AI на специфику твоего проекта. В отличие от `echo_patterns/`, она **версионируется в git**.

### Инициализация

```bash
python echo_setup.py agents
```

Создаёт шаблоны:

```
agents/
  style/
    global.md        ← именование, форматирование, импорты
    commands.md      ← стиль CLI-команд
  arch/
    overview.md      ← модули, зависимости, поток данных
  context/
    project.md       ← цель проекта, бизнес-правила
```

### Использование

Заполнил шаблоны → загружай в сессию:

```
/load style @agents/style/     ← вся папка style → один паттерн
/load arch  @agents/arch/

use style
что делает эта функция? — ИИ уже знает стиль кода проекта
```

Паттерны из `agents/` можно загружать при каждой сессии — они лёгкие (`.md` файлы).

## Git-интеграция

### Настройка (одноразово)

```
python echo_setup.py
```

Скрипт:
1. Создаёт `.echo.json` — фиксирует модель и бэкенд для команды
2. Добавляет `echo_patterns/` в `.gitignore`
3. Устанавливает post-commit хук (опционально)

### Post-commit хук

После каждого коммита хук автоматически обновляет паттерны для изменённых файлов.

**Как это работает:**

```
git commit -m "fix auth middleware"
   ↓
.git/hooks/post-commit запускается автоматически
   ↓
Определяет изменённые файлы: src/auth/middleware.py
   ↓
python echo_poc.py --refresh-file src/auth/middleware.py
   ↓
Все паттерны, использующие этот файл, пересоздаются
```

**Ручное использование флагов:**

```bash
# Обновить паттерны для конкретного файла (без запросов)
python echo_poc.py --refresh-file src/auth.py

# Проверить актуальность всех паттернов
# Выходной код 0 = всё ok, 1 = есть устаревшие
python echo_poc.py --status
```

`--status` удобно использовать в CI:

```yaml
# .github/workflows/check.yml
- name: Check Echo patterns
  run: python echo_poc.py --status || echo "Паттерны устарели — обновите через echo_poc.py"
```

### Структура путей (git-aware)

```
echo_patterns/
  <repo>/
    <branch>/
      auth.pkl          ← паттерн ветки main
      auth.json
    feature-auth/
      auth.pkl          ← паттерн ветки feature/auth
      auth.json
```

Та же структура — ключ S3 для синхронизации между командой:
```
s3://your-bucket/echo-patterns/<repo>/<branch>/<name>.pkl
```

## Автообновление паттерна

Если паттерн создан из файла (`@файл`), перед каждым `ask`/`peek` система проверяет — не изменился ли исходник:

```
⚠️  Файлы изменились с момента создания паттерна 'auth':
    • src/auth.py
   Пересоздать паттерн? [y/N]
```

`y` — автоматически перечитывает файл, пересоздаёт паттерн, затем отвечает.
`N` — продолжает со старым паттерном.

Паттерны из inline-текста (`context auth "..."`) не отслеживаются.

## Конфигурация

**echo_poc.py** (Transformer):

| Параметр | По умолчанию | Описание |
|---|---|---|
| `MODEL_PATH` | `models/Qwen3-8B-GGUF/...` | Путь к GGUF |
| `N_CTX` | `8192` | Контекстное окно (токены) |
| `N_GPU_LAYERS` | `-1` | Слоёв на GPU (-1 = все) |
| `MAX_TOKENS` | `512` | Максимум токенов в ответе |

**echo_rwkv.py** (RWKV):

| Параметр | По умолчанию | Описание |
|---|---|---|
| `MODEL_PATH` | `models/rwkv/...` | Путь к GGUF |
| `N_CTX` | `1024` | Буфер генерации (не лимит истории) |
| `CHUNK_SIZE` | `512` | Размер чанка при обработке |
| `N_GPU_LAYERS` | `-1` | Слоёв на GPU (-1 = все) |
| `MAX_TOKENS` | `512` | Максимум токенов в ответе |
